# Prisma ORM Migration — Design Spec

**Date:** 2026-05-19
**Status:** approved
**Supersedes:** ADR-001 (SQLite over PostgreSQL — decision retained; better-sqlite3 driver replaced by Prisma)

## Motivation

Replace the manual better-sqlite3 + raw SQL repository layer with Prisma ORM to standardize database operations across all feature slices. The project is in early Phase 1 (one slice built) so the migration cost is minimal.

## Architecture

```
Before:  Service → Repository → better-sqlite3 (sync, Database.Database)
                              → Manual .sql migrations + migrate.ts
                              → Hand-written types with `as T` casts

After:   Service → Repository → PrismaClient (async)
                              → prisma migrate dev/deploy
                              → Types generated by prisma generate
```

## Changes by layer

| Layer | Before | After |
|---|---|---|
| `db/index.ts` | Creates `better-sqlite3` instance, sets WAL/FK pragmas | Creates `PrismaClient` singleton |
| `db/migrate.ts` | Reads `.sql` files, runs them, tracks in `migrations` table | Removed — replaced by `npx prisma migrate deploy` |
| `db/migrations/*.sql` | Raw SQL migration files | Replaced by `prisma/migrations/` (auto-generated) |
| Repository | Takes `Database.Database`, sync methods | Takes `PrismaClient`, async methods |
| Service | Some sync calls to repository | All repository calls become awaited async |
| Type files | Hand-written interfaces in slice | Imports from `@prisma/client` |

## Schema

```prisma
datasource db {
  provider = "sqlite"
  url      = "file:../vehicles.db"
}

generator client {
  provider = "prisma-client-js"
}

model Vehicle {
  id           Int              @id @default(autoincrement())
  fipeCode     String           @unique @map("fipe_code")
  vehicleType  String           @map("vehicle_type")
  brand        String?
  model        String?
  favorited    Int              @default(0)
  fetchedAt    DateTime         @default(now()) @map("fetched_at")
  updatedAt    DateTime         @updatedAt @map("updated_at")
  years        VehicleYear[]
  scrapes      ScrapingDetail[]

  @@map("vehicles")
}

model VehicleYear {
  id             Int       @id @default(autoincrement())
  vehicleId      Int       @map("vehicle_id")
  yearCode       String    @map("year_code")
  yearLabel      String    @map("year_label")
  price          String?
  fuel           String?
  referenceMonth String?   @map("reference_month")
  fuelAcronym    String?   @map("fuel_acronym")
  fetchedAt      DateTime? @map("fetched_at")
  priceUpdatedAt DateTime? @map("price_updated_at")
  vehicle        Vehicle   @relation(fields: [vehicleId], references: [id], onDelete: Cascade)

  @@map("vehicle_years")
}

model ScrapingDetail {
  id                 Int      @id @default(autoincrement())
  vehicleId          Int      @map("vehicle_id")
  sourceUrl          String   @map("source_url")
  engine             String?
  powerHp            String?  @map("power_hp")
  torque             String?
  transmission       String?
  fuelType           String?  @map("fuel_type")
  consumptionCity    String?  @map("consumption_city")
  consumptionHighway String?  @map("consumption_highway")
  rawData            String?  @map("raw_data")
  scrapedAt          DateTime @default(now()) @map("scraped_at")
  vehicle            Vehicle  @relation(fields: [vehicleId], references: [id], onDelete: Cascade)

  @@map("scraping_details")
}

model Job {
  id        Int      @id @default(autoincrement())
  type      String
  payload   String
  status    String   @default("pending")
  attempts  Int      @default(0)
  error     String?
  createdAt DateTime @default(now()) @map("created_at")
  updatedAt DateTime @updatedAt @map("updated_at")

model CronRun {

  @@map("cron_runs")
}
```

## Repository layer changes

- All methods become async (return `Promise<T | null>` instead of `T | null`)
- `db.transaction(fn)` → `db.$transaction([...])` for batch writes
- Queries use the generated Prisma client API: `db.vehicle.findUnique({ where: { fipeCode } })`
- No more `as T` type assertions — types are generated from the schema
- Constructor still accepts the db instance via DI — same pattern, different type

## Service layer changes

- Repository calls get `await` (services are already async)
- DB types imported from `@prisma/client` instead of slice-local types
- API response types (`SearchResponse`, `YearDetailResponse`) stay in the slice
- Error handling, route delegation, and DI wiring unchanged

## Migration plan

1. Delete `backend/src/db/migrations/` and `backend/vehicles.db`
2. Install `@prisma/client` (dep) and `prisma` (dev dep)
3. Create `backend/prisma/schema.prisma` with the schema above
4. Run `npx prisma migrate dev --name init`
5. Rewrite `backend/src/db/index.ts` as PrismaClient singleton
6. Delete `backend/src/db/migrate.ts`
7. Rewrite `VehicleSearchRepository` with Prisma queries
8. Rewrite `VehicleSearchService` with await on repo calls
9. Delete `vehicleSearch.types.ts` (DB types), keep response types
10. Update tests to use Prisma-generated types
11. Update `package.json` migrate script to `npx prisma migrate deploy`
12. Run lint + tests, verify app starts and search works

## Known limitation

Prisma's SQLite provider does not support `CHECK` constraints. The `vehicle_type CHECK(IN ('cars', 'trucks', 'motorcycles'))` moves to Zod validation (already in `vehicleSearch.validator.ts`). App-layer validation is the correct place for this.
